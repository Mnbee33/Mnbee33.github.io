### エピソード

**チームの品質向上のために、試行錯誤**  

設計工程が始まって間もない頃、プロパーレビュー（一次請会社の社員による設計書レビュー）の指摘の多さがチームの課題だった。設計書の作成ルール違反が指摘の半数以上を占め、レビューアからも成果物の品質に関して苦言を呈される程であった。

なぜ設計者はルール違反をしてしまうのか、チーム内のレビューアは違反を見つけられないのか、二つの原因に当たりをつけた。

1. <u>元々、ルールの数が多い。</u>１機能につき、10種類以上のドキュメントが必要。１ドキュメントあたりのルールも片手では収まらない。ルールブック・サンプルを見ながら設計書を作成・確認しなくてはならないが、目視ではどうしても見落としが発生していた。

2. <u>事前に提示された作成ルールが絶対ではない。</u>元請け会社レビュー・プロパーレビュー（他の機能の分も含む）の指摘から、設計書の新たな改善点が発見される。レビューを重ねるごとに、ルールが増える。  追加されたルールだけで30個超え、ということも。

２つの原因を解決するために、対応策を考え、実行した。
- 設計書のチェックツールを自作
  - 設計書の作成ルールが膨大なことへの対策。ルールどおり作成できているか、ツールでまとめてチェックできるようにした。
  - 目視では検知が不十分だったポイントも、漏れなくチェックできるようになった。
  - チェックにかかる時間も20分から1〜2分まで短縮。
- ルール追加・変更の周知
  - 朝一番に（他チームの成果物も含めて）レビュー記録票を確認。設計書作成のルールの追加や変更があった場合、朝会とメールでチーム全体に周知した。
- チェックリストを作成
  - ルールの総数が、設計者もチーム内のレビューアも記憶できる範囲を超えていたため、作成ルールを一覧化した。
  - 設計者には、取り込み漏れがないかのセルフチェックシートとして、チーム内のレビューアには、レビューの観点表として確認してもらった。

施策の甲斐もあり、プロパーレビューで作成ルール違反の指摘は大幅に減少した。指摘ゼロの設計書も提出できるようになった。

### エピソード
{: .no_toc }

新しい技術への学習ステップ

当初、顧客から指定されたフレームワーク・ライブラリの使用経験のあるメンバは、自分を含め、チームにいなかった。円滑にプロジェクトを進めていくためには、少しでも早く、新しい技術に馴染むことが不可欠だった。

- 技術の学習と実践
  - 時間を見つけて、フレームワークの公式ドキュメントを地道に翻訳。パラメータの受け渡し方法が複数パターンあることを事前に知った。画面遷移する場合、非同期通信で表示データを取得する場合など、追加する機能に応じたデータの受け渡し方法を選択し、実装できた。
  - 実際にライブラリを使って作成された、過去のソースコードを解析。APIドキュメントと照らし合わせながら実例を読み解くことで、理解速度が向上した。
  - 製造工程が本格化する前に、サンプルプログラムを作成。導入候補となるライブラリやプラグインを試すことで、使用方法を学ぶと同時に、要件どおりの機能が実現できるかを検証した。

- 自身の理解が進んだら、メンバへ伝導
  - 開発環境の構築や本番環境へのデプロイ等、チームで初めての作業を行なった際に、ガイドラインを併せて作成した。自分が躓いてしまった作業は、対応方法の理由も含めて詳細に記載し、他メンバ（未来の自分を含む）が同じ轍を踏んで苦労するのを防いだ。
  - 既存ソースコードの解析結果やガイドラインの和訳等、勉強した内容をドキュメントに整理した。他メンバの理解の手助けとなるよう、作成したドキュメントはチーム内に展開した。ドキュメントの作成過程で、自分自身の考え・知識も整理できた。

クリーンなプログラムへの憧れと目覚め

プログラム設計の師匠とも呼ぶべき上長（兼チームリーダー）の配下となる。オブジェクト指向、リファクタリング、デザインパターン等について、上長からの教育を受ける。

それまでは、ただ正しく動かすことだけを考え、プログラムを設計し、作成してきた。後々の試験、保守、改修で、どれだけの負債を抱えることになるかも考えず。

後工程での苦労の元凶は、自分自身が作り上げたプログラムであったことに気付かされた。「後々変更しやすいプログラム」「作っていても、読んでいても気持ちが良いプログラム」への憧れが強くなる。以降、書籍を購入し、業務外でも自学自習し始める。

## 生命保険 社内Webシステム保守開発

半年に１度の法改正に合わせて、システムを改修する。本番稼働するシステムで障害が発生した場合は、原因の調査と修正も行う。チームの規模は20名程度。

### 参画期間
{: .no_toc }

2015年11月〜2017年3月

### 担当機能
{: .no_toc }

- チームの担当は、請求案内機能。顧客が請求した生命保険料の他に、支払可能な保険契約が存在するかチェックする。支払可能な契約が存在した場合、顧客へ支払請求を促す案内状を出す。
  - 契約の候補を抽出するバッチ機能と、案内状の送付を承認するための画面機能に分かれる。
  - 各メンバは特定の担当機能は持たない。改修規模に応じて、適宜担当が割り当てられる。
- 主に画面側の設計〜結合試験を担当。
  - 単体・結合試験を担当することが多かった。複数人で試験する結合試験では、メンバが円滑に試験が進められるよう、手順書やエビデンス整形ツールを作成したり、メンバの作業実績の確認と振り分けをしたりした。
  - 対応範囲の大きかった案件では、日次バッチも結合試験で検証。
  - 結合試験を通して、サブシステム内のほぼ全ての機能について把握。
